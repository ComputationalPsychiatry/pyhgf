# The DeepNetwork architecture

This page is the implementation companion to the
[theory page](0.5-Deep_networks_theory.md), which derives the weight
updates, and the [deep-learning tutorial](0.7-Deep_learning.ipynb), which walks
through a worked classification example. It explains how PyHGF assembles and
trains deep models — the core `DeepNetwork` class, the mixed-pipeline machinery
that builds larger models, and the general recipe for porting any architecture,
up to a full Transformer trained **without backpropagation** — for a reader who
has never used PyHGF before.

---

## 1. The main idea

**How training normally works.** A deep learning model makes a guess, the loss measures how wrong it was, and *backpropagation* (i.e.,one global procedure sweeping backward through the entire network with calculus) works out how much every weight contributed to the mistake. Nothing is local: every weight update needs information computed from every other part of the model.

**How PyHGF trains instead.** PyHGF networks are made of nodes that hold *beliefs*: each belief is a guess (a mean) plus a confidence in that guess (a **precision**, where high precision means very confident). Every layer predicts the layer below it; the difference between a prediction and what actually happened is a **prediction error**. Learning is local: each connection adjusts its weights using only the prediction error on one side and the activity on the other, weighted by confidence. No part ever sees the whole network, and no automatic differentiation runs anywhere in training. PyHGF is *single-sweep* predictive coding:
predict once, correct beliefs once, nudge weights once per batch. It is not
the textbook settle-to-equilibrium variant, where beliefs relax through many
iterations before any weight changes.

The two schemes provably coincide (the theorem is derived in the [theory page](0.5-Deep_networks_theory.md), §3): in a pinned configuration, one PyHGF learning step reproduces backpropagation's weight updates to better than 1e-5 relative error, and a full Transformer trained this way follows a backprop twin's loss curve to four decimal places. What PyHGF adds beyond parity is the confidence machinery: every belief knows how reliable it is, and updates are weighted accordingly.

## 2. A `DeepNetwork`: layers of beliefs

### 2.1 Building one

```python
import jax
import optax
from pyhgf.model import DeepNetwork

net = (
    DeepNetwork()
    .add_layer(size=8)                              # output layer (bottom)
    .add_layer(size=16, coupling_fn=jax.nn.gelu)    # hidden layer
    .add_layer(size=8)                              # input layer (top)
)
```

Three conventions to know:

- **Input on top, output at the bottom.** Layer 0 is the *output* layer
  (where observations are clamped); the last layer added is the *input*
  layer (where predictors are clamped). Predictions flow downward.
- **The nonlinearity belongs to the predicting layer.** A layer's
  `coupling_fn` is applied when that layer predicts the one *below* it. The
  network above therefore computes `W1 @ gelu(W2 @ x + b2) + b1` — exactly a
  standard `Linear → GELU → Linear` block, with the hidden nodes holding the
  pre-activation values.
- **Biases are constant nodes.** Each layer can carry an always-on constant
  node (`add_constant_input=True`, the default); the bias lives as an extra
  column of the weight matrix, added linearly — the same convention as every
  standard library.

### 2.2 The three sweeps

One learning step is three passes over the layers:

1. **Prediction sweep** — clamp the input `x` on the top layer; every layer
   predicts the one below it, top to bottom.
2. **Update sweep** — clamp the observation `y` on the bottom layer; compute
   prediction errors and correct every belief, bottom to top, each
   correction weighted by confidence.
3. **Learning sweep** — nudge every weight from its local prediction error
   and the activity on its other side.

### 2.3 Two ways to feed data

**As a time series** (`net.fit(x, y, ...)`): samples are scanned one at a
time, and the confidences *carry over* — each prediction reads the previous
posterior precision, so the network's confidence adapts as the series
unfolds. This is PyHGF's native filtering mode.

**As a batch** (`net.batch_update(x, y, ...)`): samples are exchangeable
(e.g. token positions), so every sample is processed from the *same* state —
same weights, same confidences — in parallel, and the per-sample updates are
averaged and applied once. The batch counts as a single observation, which
makes the step invariant to repeating the batch.

```python
net.batch_update(
    x_batch, y_batch,                # (batch, n_features) each
    optimizer=optax.adam(3e-3),      # any optax optimizer, applied locally
    update_confidences=False,        # True: confidences also adapt per batch
)
net.input_errors                     # (batch, n_input) — error at the input,
                                     # for whatever sits behind this network
```

Two flags control what learns: `optimizer=None` freezes the weights;
`update_confidences=False` freezes the confidences (used whenever an exact
comparison against backpropagation is wanted). `net.input_errors` is the
confidence-weighted prediction error routed back to the input layer — the
message that lets networks be chained (§3).

## 3. Mixed pipelines: assembling bigger models

Large models are built from **parts** (`pyhgf.model.hybrid`). The part
objects *declare* the model — which slots learn, which are frozen, how they
nest — and every part contributes the same two things to the compiled
training step:

- forward: compute my output from my input;
- backward: receive the error at my output, learn from it if I have
  weights, hand the error at my *input* to the part behind me.

The arrays passed between parts are *descent errors* — the gradient of the
loss with respect to that signal, the same object a backprop library would
hand around.

Two kinds of parts exist:

- **Learning parts** — `DeepNetworkAdapter(net, optimizer=...)` wraps a
  `DeepNetwork`. It is the single place where the pipeline's error
  convention meets PyHGF's (observed-minus-predicted) convention: the error
  enters by clamping `output − error` as the observation the network should
  have produced, and leaves as the negated `input_errors`.
- **Frozen parts** — fixed calculations with nothing to learn, which only
  *translate* errors using a hand-derived formula (no autodiff, not even
  locally; autodiff appears only in tests, as an oracle to check the
  formulas). Ready-made: `linear_adapter` (error multiplied back through the
  weights), `gelu_adapter` (error times the slope at the cached input),
  `layer_norm_adapter` (undo the rescaling, subtract the row-mean and the
  overlap with the normalised input).

Two combinators wire parts together: `PCSequential` (a chain) and `Residual`
(the shortcut junction `output = input + branch(input)`; backward, the error
is copied into both routes and the two messages add). A Transformer
sub-block is one expression:

```python
from pyhgf.model import (
    DeepNetworkAdapter, PCSequential, Residual,
    from_feedforward, layer_norm_adapter,
)

block = Residual(PCSequential([
    layer_norm_adapter(ln),                              # frozen
    DeepNetworkAdapter(                                  # learns
        from_feedforward(fc1, fc2),                      # Linear→GELU→Linear
        optimizer=optax.adam(3e-3),
    ),
]))
```

Backwards through one full Transformer block, the error travels like this —
no global procedure, each arrow a local computation:

```{mermaid}
flowchart TD
    EIN["error arriving from the part in front<br/>(nearer the output)"] --> C2{"copy"}
    C2 -->|straight route| P2["unchanged"]
    C2 -->|branch route| FF["feed-forward (PyHGF):<br/>nudges its weights,<br/>emits error at its input"]
    FF --> LN2["LayerNorm formula:<br/>translates the error"]
    LN2 --> A2(("add"))
    P2 --> A2
    A2 --> C1{"copy"}
    C1 -->|straight route| P1["unchanged"]
    C1 -->|branch route| WO["O table (PyHGF):<br/>nudge + emit"]
    WO --> MIX["attention mixing (frozen):<br/>re-routes error across positions"]
    MIX --> QKV["Q, K, V tables (PyHGF):<br/>each nudges + emits;<br/>their three messages add"]
    QKV --> LN1["LayerNorm formula"]
    LN1 --> A1(("add"))
    P1 --> A1
    A1 --> EOUT["error handed to the block behind"]
```

### Running a pipeline: one compiled program per step

The part objects only declare the model; `FusedPipeline`
(`pyhgf.model.fused`) runs it. Each training step — the forward walk, the
error at the output, every part's local learning step — is staged into a
**single compiled program**, so nothing crosses a compilation boundary
inside a step and the per-sample forward states are already in hand when
each part's update needs them. Usage:

```python
from pyhgf.model import FusedPipeline

pipeline = FusedPipeline(model, lambda probs, ids: probs - jax.nn.one_hot(ids, vocab))
for x, y in batches:
    output, input_error = pipeline.step(x, y)   # one compiled call
probs = pipeline.predict(x_eval)                # forward pass only
pipeline.merge()   # write the trained state back onto the part objects
```

The `error_fn` given at construction forms the descent error at the output
*inside* the program (for a `HybridGPT`, `step` and `predict` take integer
token ids and the one-hot encoding also happens in-trace); when omitted,
it defaults to `output - target`, the squared-error gradient. `predict`
runs the forward pass alone, for evaluation and generation. The part
objects do not advance while the executor runs; `merge()` writes the held
state back onto them for inspection or checkpointing. Measured on the
notebook-scale Transformer (CPU): the whole training step runs within
~1.2–1.5× of an equivalent fused backprop step, and compiles once per
input shape in a few seconds.

## 4. The general recipe: porting any architecture

Any feed-forward deep learning model maps onto a mixed pipeline with five
rules.

**Rule 1 — every weight matrix becomes a small PyHGF network.** The
`pyhgf.model.transplant` converters build them directly from trained (or
freshly initialised) Equinox layers, weights carried across so both sides
compute the identical function:

| Converter | Standard component | Notes |
| --- | --- | --- |
| `from_linear(linear)` | any weight table (projections, output head) | bias folded in as an extra column |
| `from_feedforward(fc1, fc2)` | `fc2(act(fc1(x)))` block | `fc1`'s bias lands *inside* the activation, on the input→hidden matrix |
| `from_embedding(emb)` | lookup table | a lookup is a one-hot vector times the table; feed one-hot rows |

**Rule 2 — weightless calculations become frozen parts.** Activations,
normalisations, reshapes: each needs only a hand-derived backward formula,
verified in tests against autodiff of the same forward (autodiff as test
oracle, never in training).

**Rule 3 — multi-input operations become composites.** An operation that
reads several streams cannot be a single-input part; wrap it and its
neighbours into one part exposing the standard contract. The attention
composite (`pyhgf.model.MultiHeadAttention`) is the pattern: it holds four
single-input parts (Q, K, V, O — frozen or learning, interchangeably) around
the weight-free mixing, whose backward re-routes errors across token
positions using the cached attention pattern.

**Rule 4 — non-feature inputs sit outside the contract.** Token ids are
integers, not features: embeddings live at the pipeline boundary, either as
frozen lookup tables or as learning parts fed one-hot rows.

**Rule 5 — classification outputs are belief nodes; the error is born, not
injected.** Clamping the truth onto the output layer makes the
classification error arise from the model's *own* beliefs. Two kinds serve
two tasks:

- `kind="categorical"` — **multi-class (the default choice)**: the layer is
  one softmax choice across its nodes; clamping the one-hot truth yields
  the raw residual `one_hot − softmax`, which is exactly the cross-entropy
  gradient, so weight updates and the emitted input error match a softmax
  backprop head. (The layer's unit-precision convention makes every
  existing kernel produce this with no special-casing.) Measured on the
  Transformer: at its tuned rate the categorical-head learner matches
  backprop-CE in every condition — 0.969 vs 0.965 clean, 0.900 ± 0.006 vs
  0.896 under 25% label noise.
- `kind="binary"` — **multi-label**: one independent yes/no belief per
  node, matching a sigmoid-cross-entropy head. Using it for a multi-class
  task costs measurable accuracy under label noise (~4 points at 25% noise
  in the same protocol), because random wrong labels hit each independent
  belief separately.

### 4.4 Verifying a port

Two checks, in order:

1. **The wiring gate (learning-free).** Transplant every weight from the
   reference model, freeze every slot, and require the pipeline to
   reproduce the reference outputs to floating-point precision. One
   assertion validates every adapter, orientation, and bias placement.
2. **Learning parity.** Put the learning parts in the **parity recipe** —
   volatility levels frozen (`volatility_parent=False`, low
   `tonic_volatility`), high prior confidence (`precision=1e4`,
   `expected_precision=1e4`) on hidden and input layers, unit confidence on
   the observed output layer, `learning_kind="precision_weighted"`, and
   `update_confidences=False` — and one training step matches
   backpropagation's weight changes (gate tolerance: 1e-2 norm-relative in
   float32; ~1e-5 in float64). The recipe works because the hidden layer's
   belief shift is its routed error *divided by* its posterior confidence,
   and the confidence-weighted update multiplies that confidence back in —
   the two cancel exactly.

### 4.5 Choosing the optimizer and the learning rate

A ported model brings a tuned learning rate with it — and that rate was
tuned for backpropagation's dynamics, not PyHGF's. The two learners sit on
*shifted stability frontiers*: PyHGF's updates are more damped (the belief
corrections absorb part of each error before the weights see it), so at the
same nominal rate it takes effectively smaller steps. Measured on the
Transformer task with plain SGD and a matched objective: backpropagation's
accuracy peaked at rate 0.6 and collapsed at 0.65, while the PyHGF learner
peaked at 0.7 and stayed stable well past that — and at each learner's own
optimum the two were statistically indistinguishable. Three practical
rules follow:

- **A backprop-tuned rate is a lower bound, not a transplant.** Expect the
  PyHGF optimum to sit noticeably higher (~15–20% on this task) with a
  wider stable range (~35% here). These multipliers were measured at one
  scale — treat them as a starting point.
- **Benchmark at per-learner tuned rates, with matched objectives.**
  Rate-matched comparisons systematically understate PyHGF; at a shared
  rate below its optimum it trailed by up to 9 accuracy points that vanish
  entirely once tuned.
- **When in doubt, use Adam.** Adam rescales every weight's step from the
  gradients it observes, which removes the calibration problem in practice:
  the full-Transformer run in notebook 0.8 matches backprop-Adam accuracy
  with the same nominal rate, untuned. Plain SGD remains the right choice
  for parity experiments, where an optimizer with internal state would blur
  what is being compared.

To see where a rate is off, `step_report` collects each learning part's
last applied weight-change norm next to the error it received — the
quantities are recorded inside the compiled step itself:

```python
from pyhgf.model import step_report

pipeline.step(x, y)
for entry in step_report(pipeline):
    print(entry["part"], entry["update_norm"], entry["error_norm"])
```

Parts whose updates are tiny relative to their errors are under-driven;
parts whose updates rival their weight scale are near the stability edge.

One stability bonus worth knowing: the wider stable range is not just
tolerance for sloppy tuning. At tuned rates on clean data,
backpropagation's training collapsed on half the seeds while the PyHGF
learner converged on all of them at an even higher rate — the damping buys
reliability.

### 4.6 Beyond transformers: what other architectures would need

- **Convolutional networks.** A convolution is a weight-shared linear map
  over patches: one small PyHGF network applied to every patch (an
  im2col-style reshape is a frozen part), with the per-patch weight nudges
  averaged — the same batch-averaging machinery `batch_update` already uses
  across positions. Pooling and padding are frozen parts with simple
  backward formulas.
- **Recurrent networks.** The recurrent weight matrix is one PyHGF network
  applied at every time step; the unrolled chain is a pipeline whose depth
  is the sequence length. Notably, PyHGF's *sequential* mode — confidences
  carrying over step by step — is the natural fit here rather than a
  workaround.

Neither is implemented; porting one end-to-end is future work.

## 5. Worked example: the Transformer

The full architecture, forward from token ids to the output error — every
weight table a PyHGF learner (green), every weightless calculation a frozen
part (orange). This is the character-level GPT of the two notebooks: 64
feature dimensions, 4 attention heads, a 128-wide feed-forward hidden layer,
two stacked blocks. A **copy** junction splits a shortcut from its branch; an
**add** junction merges them back (the residual connection).

```{mermaid}
flowchart TB
    IDS["token ids<br/>(batch, seq) integers"]
    IDS --> TOK["token embedding<br/>one-hot · table → (·, 64)"]
    IDS --> POS["position embedding<br/>one-hot · table → (·, 64)"]
    TOK --> E0(("add"))
    POS --> E0
    E0 --> BIN["block input<br/>(batch, seq, 64)"]

    subgraph BLOCK["one Transformer block — repeated ×2"]
        direction TB
        BIN2["input"] --> S1{"copy"}
        S1 -->|shortcut| A1(("add"))
        S1 -->|branch| LN1["LayerNorm n1"]
        LN1 --> Q["Q table<br/>64→64"]
        LN1 --> K["K table<br/>64→64"]
        LN1 --> V["V table<br/>64→64"]
        Q --> MIX["attention mixing — 4 heads<br/>scaled dot-product · causal mask<br/>softmax · blend values<br/>(no weights)"]
        K --> MIX
        V --> MIX
        MIX --> O["O table<br/>64→64"]
        O --> A1
        A1 --> S2{"copy"}
        S2 -->|shortcut| A2(("add"))
        S2 -->|branch| LN2["LayerNorm n2"]
        LN2 --> FF["feed-forward<br/>Linear 64→128 · GELU · Linear 128→64"]
        FF --> A2
        A2 --> BOUT["output"]
    end

    BIN --> BIN2
    BOUT --> NF["final LayerNorm"]
    NF --> HEAD["output head<br/>64→vocab · categorical softmax beliefs"]
    HEAD --> ERR["cross-entropy error<br/>one_hot(target) − softmax<br/>(born at the output layer)"]

    LG["learns via PyHGF"]
    LF["frozen · hand-derived error"]

    classDef learn fill:#cfe8cf,stroke:#2e7d32,color:#111;
    classDef frozen fill:#fde9c8,stroke:#b26a00,color:#111;
    class TOK,POS,Q,K,V,O,FF,HEAD,LG learn;
    class LN1,MIX,LN2,NF,LF frozen;
```

Reading it as the pipeline of §3: each block is two `Residual` shortcuts in a
`PCSequential` — the first wraps `LayerNorm → MultiHeadAttention`, the second
wraps `LayerNorm → feed-forward`. The four attention tables (Q, K, V, O) sit
as single-input parts around the weight-free mixing; the two embeddings live
at the pipeline boundary, fed one-hot rows; the head is a categorical output
layer, so clamping the one-hot target makes the cross-entropy error arise
from the model's own beliefs (§4, rule 5). Backward, the error at each `add`
is copied into both routes and the two messages sum again — the block-level
error path is the graph in §3.

`pyhgf.model.hybrid_from_gpt` assembles the whole model from a reference
Equinox GPT; every slot defaults to frozen-at-the-given-weights, and any
subset can learn:

```python
from pyhgf.model import hybrid_from_gpt

wiring_check = hybrid_from_gpt(gpt)          # everything frozen
ff_swap = hybrid_from_gpt(gpt, ff_parts=ff)  # only feed-forwards learn
full = hybrid_from_gpt(                      # every weight learns
    gpt, ff_parts=ff, attention_parts=attn,
    head_part=head, token_part=tok, position_part=pos,
)
```

The full story, with executed results, is
`docs/source/notebooks/0.8-Transformers.ipynb`: the exactness gates live in
`tests/test_transformer.py` (the wiring gate passes at 5.7e-6; the
feed-forward swap follows its backprop twin's loss curve to four decimals);
the full model — every weight learning through PyHGF in the exact
backprop-parity mode (`learning_kind="precision_weighted"`, §4.4), categorical
head, a local Adam per part — trains
from scratch to ~97% next-character accuracy in 400 steps and generates
recognisable text. On wall-clock
at this scale (CPU), the executed notebook measures **~31 ms per step against
~24 ms for backprop — a factor of ~1.3** (CPU wall-clock at this scale is
noisy; both timed loops include their one-off compilation). Both learners run one
compiled program per
step; the remaining difference is PyHGF's genuinely larger per-step
computation (the confidence side), not implementation overhead.

## Why it works

That a *silent-interior* predictive-coding network reproduces backpropagation node-for-node is a theorem, not a coincidence of this recipe. The derivation — and when predictive coding instead does something backpropagation cannot — is in the [theory page](0.5-Deep_networks_theory.md), §3.
